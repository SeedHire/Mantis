Model Router — 3-Layer Intent Classification

The router classifies every message into one of 7 tiers with 97.7% accuracy, selecting the optimal model for latency and capability. Three classification layers run in sequence; the first confident match wins.

Layer 1: Structural Rules (0ms)

Hard-coded rules that fire before any scoring:

  • Message contains an image attachment → Vision
  • Terminal error output detected → Code
  • Explicit tier override via --model flag → forced tier

Layer 2: Accumulated Keyword Scoring (0ms)

Each keyword in the message contributes a weighted score to one or more tiers. The tier with the highest accumulated score wins.

TierWeightExample Keywords
Max1.5redesign system, full-stack, migrate entire
Reason1.2explain why, analyze, compare tradeoffs
Heavy1.3refactor, deep analysis, optimize
Fast0.9quick fix, rename, small change
Code0.8implement, write function, add endpoint
Trivial0.8what is, define, yes/no

Dampeners

Two dampeners adjust scores after keyword accumulation:

  • Question-form: Trailing ? → Heavy × 0.4, Code × 0.7, Reason × 1.3
  • Short message: ≤4 words → Heavy × 0.35, Max × 0.35

Confidence Calculation

confidence = 0.60 + 0.35 × (bestScore / totalScore), capped at 0.95

If confidence ≥ 0.82, the tier is accepted. Otherwise, Layer 3 fires.

Layer 3: Embedding kNN Fallback

When keyword scoring is ambiguous (confidence < 0.82), the message is embedded and compared against 100+ labeled examples using k=5 majority vote with score-weighted tally.

  • 100+ labeled (query, tier) pairs covering all 7 tiers
  • LRU cache (128 entries, random eviction) avoids re-embedding repeat queries
  • Protected by sync.RWMutex for concurrent access
Reference: RouteLLM paper — routing efficiency. The 3-layer approach keeps 98%+ of classifications at zero latency cost (Layers 1-2), only invoking embedding inference for genuinely ambiguous inputs.

Why 7 tiers instead of one large model? A 4B parameter model answers "what is a goroutine?" in ~200ms. Routing that to a 70B reasoning model wastes 3-5 seconds and GPU memory. The 7-tier system optimizes for the distribution of real developer queries: ~40% trivial/fast, ~35% code, ~25% reason+heavy+max.

Context Engine — Multi-Signal File Scoring

The context bundler selects and orders files for the model's context window using a multi-signal scoring system. Each file in the project receives a composite score, and the top-scoring files are packed into the token budget.

Scoring Signals

SignalScoreLogic
Graph depth0–5Closer to the target symbol in the dependency graph = higher
File size0–3Smaller files score higher (more likely to be focused modules)
Recency0–2int(2.0 / (1 + daysSince/30.0))
Test co-location+3 / −4Same base name as target → +3; other test files → −4
Same package+2Files in the same package as the target
Type boost+1Type definition files (interfaces, structs)
Vendor penalty−10Third-party / vendor / node_modules
Churn bonus0–4From git temporal coupling analysis
score = depth + size + recency + test_colocation + same_pkg + type_boost + vendor_penalty + churn_bonus

Churn Coupling

Before scoring, the bundler runs intel.Temporal and CouplingFor to identify files that historically change together with the target. Coupled files receive up to +4 bonus points.

Lost in the Middle (Liu et al. 2023)

Paper: "Lost in the Middle: How Language Models Use Long Contexts" — Liu et al. 2023. LLMs attend more to the beginning and end of their context window, with significant attention degradation in the middle.

Mantis places the most relevant file last in the context bundle (reversed sort order). This ensures the highest-scoring file occupies the position with strongest model attention — immediately before the user's message.

Token Budgeting

Files are packed greedily by descending score until the token budget is exhausted. A binary-search approach is used when a file exceeds the remaining budget: it's truncated to fit, preserving the most important lines (function signatures, type definitions).

Inspiration: Sourcegraph Cody's multi-signal file scoring. Mantis extends the approach with temporal coupling data from git history, which Cody does not incorporate.

Embeddings & Hybrid Retrieval

Storage Architecture

Embeddings are stored as binary BLOBs (float32 arrays) in SQLite, achieving 4× smaller storage compared to JSON text encoding. An FTS5 virtual table provides BM25 full-text search with automatic synchronization via 3 triggers (INSERT/UPDATE/DELETE).

-- Core schema
CREATE TABLE chunks (
    id TEXT PRIMARY KEY,
    source TEXT,
    section_label TEXT,
    content TEXT,
    content_hash TEXT,
    embedding BLOB    -- float32 binary, not JSON
);

-- FTS5 virtual table
CREATE VIRTUAL TABLE chunks_fts USING fts5(
    content, content='chunks', content_rowid='rowid'
);

-- 3 sync triggers: ai_chunks_ai, ai_chunks_ad, ai_chunks_au

Hybrid Retrieval Pipeline

Search combines two ranking signals via Reciprocal Rank Fusion:

  1. BM25 (FTS5): Query is split into words, each word >2 chars is wrapped in quotes, joined with AND. Top 20 results.
  2. Cosine Similarity: Query is embedded, compared against all 200 rows via dot product on normalized BLOB vectors.
  3. RRF Fusion (k=60): Both ranked lists are merged using Reciprocal Rank Fusion.
RRF_score(doc) = ∑ 1 / (k + ranki)  for each ranking list i, where k = 60
Why RRF over learned fusion? RRF requires no training data and the k=60 constant is robust across domains (Cormack et al. 2009). For a single-user CLI tool with no labeled relevance judgments, learned-to-rank would overfit catastrophically.

Section-Aware Chunking

Each memory file type has a dedicated splitter that respects document structure:

FileSplitterStrategy
BRAIN.mdsplitBrainMDSplit on ## headers, each section is a chunk
DECISIONS.logsplitDecisionsLogSplit on timestamp-prefixed lines
REJECTED.mdsplitRejectedMDSplit on rejection entries
CONVENTIONS.mdsplitConventionsMDSplit on rule sections

Content-Hash Guard

Each chunk is SHA-256 hashed before embedding. The Add() method checks the hash against the stored value and skips re-embedding if unchanged. This makes startup re-indexing O(changed) instead of O(all), critical when IndexBrainFiles runs on every session start.

Code Indexing

Source code chunks are split at AST boundaries using tree-sitter parsers rather than character-count windows. This preserves semantic boundaries (function bodies, type definitions) and avoids splitting mid-statement.

Reference: Cline and Aider research reports 15–30% improvement in retrieval precision when using AST-based chunking over character-based splitting for code.

SWE Pipeline — 4-Stage Code Generation

Stage Flow

PLAN → CODE ∥ TESTS → TEST_LOOP
StageModelDescription
PLANTierReasonDecomposes the task into 6–10 sub-tasks with file assignments
CODETierCodeGenerates implementation with build verification
TESTSTierCodeGenerates test files (runs in parallel with CODE)
TEST_LOOPTierCodeIteratively runs and fixes failing tests

Task Scheduling

Tasks are not all parallelized. The pipeline uses a sequential-then-parallel strategy:

  • Tasks 0–1 run sequentially (project setup + shared type definitions)
  • Tasks 2+ run in parallel batches (max 3 concurrent)

Sealed Types Manifest

After sequential tasks complete, a SEALED TYPES manifest is extracted listing all shared type definitions. This manifest is injected into every parallel worker's prompt, preventing type redefinition conflicts.

Diff-Based Application

Code changes are applied using SEARCH/REPLACE edit blocks rather than whole-file rewrites:

```edit:internal/auth/service.go
<<>>SEARCH
```
  • Parser: parseEditBlocks() extracts file path, old text, new text
  • Validation: exact-once match required (not-found and ambiguous both fail)
  • Path traversal protection via safePath()
  • Fallback: if no edit blocks found, falls back to whole-file ```filepath blocks
Why SEARCH/REPLACE over unified diff? Models produce cleaner SEARCH/REPLACE output with lower error rates. Unified diff format requires precise line numbers and context lines that models frequently get wrong, leading to patch application failures.

Iterative Fix Loop

Each CODE task has a bounded retry loop (max 3 iterations):

  1. Generate code → apply edits
  2. Run autofix.Check() (build verification)
  3. If build fails, feed error back to model and retry
  4. Stuck detection: if the same build error appears twice consecutively, break immediately

Content Validation

Generated code is checked for stub ratio: the proportion of function bodies that are empty or contain only comments/TODOs. If stub ratio > 0.3, the output is rejected and re-prompted.

Multi-Agent Fan-Out

Activation Gate

ShouldRunMultiAgent = TotalFiles ≥ 4 AND distinctPackages ≥ 2

The multi-agent system only activates for high-impact changes that span multiple packages. Smaller changes use the single-model pipeline.

3-Step Execution

Orchestrate → Fan-Out → Synthesize
  1. Orchestrate: TierReason decomposes the task into per-package sub-tasks (JSON output)
  2. Fan-Out: Workers run in parallel goroutines, each executing an AgentLoop (max 5 iterations per worker)
  3. Synthesize: TierReason combines all worker summaries, prepends code blocks, produces final output

DAG-Based Planning

Paper: arXiv 2511.10037 — "Global DAG planning outperforms greedy ReAct on multi-tool tasks." The orchestrator builds a dependency map between sub-tasks, performs topological sort, and executes level-by-level in parallel rather than greedy sequential execution.

A self-critique step runs before execution: the orchestrator reviews its own plan and adjusts task boundaries, dependencies, and ordering.

Worker Communication

Workers communicate via AGENT_SCRATCH.json in the .mantis/ directory. Each worker reads shared state before starting and writes its results on completion. This avoids the complexity of real-time message passing between goroutines.

Tool Safety

The RunBash tool enforces a strict allowlist:

Allowed prefixes: go, npm, cargo, pytest, git
Output cap: 8,000 characters
Path traversal: blocked by safePath()

Iterative Test Loop

Loop Architecture

The test loop is a closed feedback cycle: run tests → parse failures → read source → prompt fix → apply → repeat.

Run Tests → Parse → Read Source → LLM Fix → Apply ↺

Cargo.tomlRustcargo test
pyproject.tomlPython/pytestpytest

Language-Specific Failure Parsers

Each runner has a dedicated parser that extracts structured TestFailure objects:

  • Go: --- FAIL: + file.go:line: patterns
  • Jest: Bullet (•) test names + at (file:line) stack locations
  • Pytest: FAILED test_file.py::test_name + E assertion lines
  • Rust: thread 'name' panicked at + test name ... FAILED
  • Fallback: Generic regex matching FAIL/ERROR lines

Source Context Injection

For each failure, the loop reads 40 lines of source code centered around the failure location. This gives the model enough context to understand the surrounding logic without overwhelming the context window.

Stuck Detection

Each set of failures produces a FailuresKey — a hash of all failure test names and messages. If the same key appears after a fix attempt, the loop stops immediately. Default max iterations: 5.

Verification & Safety

Ground Truth Hallucination Detection

Mantis builds GROUND_TRUTH.json from the live AST using tree-sitter. This file contains every exported function signature in the codebase. When the AI references a function, its signature is checked against ground truth.

  1. Extract: Tree-sitter parses all source files, extracts exported function names, parameter types, return types
  2. Filter: 40+ stop words removed (Go builtins: if, len, append, make, fmt, error, etc.)
  3. Match: Each function reference in the AI output is checked against the ground truth map
  4. Correct: FindClosest() uses string similarity to suggest the correct signature if a mismatch is found
Impact: Without ground truth checking, LLMs hallucinate function signatures ~15-20% of the time when working with large codebases. The ground truth guard reduces this to <1%.

Convention Enforcement

Every AI response is checked against .mantis/CONVENTIONS.md. The enforcement uses the same re-prompt loop pattern as hallucination detection:

  1. Parse conventions from Markdown (supports Imports, Dependencies, Naming, Architecture sections)
  2. Check AI response against each rule
  3. If violation found: inject the violated rule into the prompt and re-generate
  4. Loop until compliant (same bounded retry as hallucination loop, ~lines 519–556 of repl.go)

Path-Scoped Rules

Conventions support [path: glob] directives, so rules can be scoped to specific directories or file patterns. A rule like [path: handlers/**] only applies when the AI is working on files matching that glob.

Import Extraction

The extractForbiddenImport() function handles 13+ rule phrasings for import restrictions (e.g., "never import from", "do not use", "forbidden import", "must not import", etc.).

Inspiration: Constitutional AI (Anthropic) — the output validation loop mirrors the constitutional revision pattern: generate → critique → revise → repeat.

Graph Intelligence

AST Dependency Graph

The graph is stored in SQLite (.mantis/graph.db) and built by parsing source files with tree-sitter. Multi-language support: Go, TypeScript, Python, Rust, Java.

-- Core tables
files    (path, language, last_modified)
symbols  (name, kind, file_path, line, column)
edges    (from_symbol, to_symbol, kind)  -- import, call, type_ref

PageRank for Symbol Importance

Reference: Aider's repo-map uses PageRank over call-graph edges. Their benchmark showed improvement from 20% → 61% on code navigation tasks when the model receives a PageRank-weighted map instead of a flat file list.

Mantis runs PageRank with:

  • Damping factor: 0.85 (standard)
  • Iterations: 20 (convergence typically at 12–15)
  • Result: each symbol gets an importance score used for context prioritization

Temporal Analysis

Churn Score

churn = (linesAdded + linesDeleted) / totalCommits

Coupling Ratio

coupling(A, B) = coChanges(A, B) / min(commits(A), commits(B))

A coupling ratio > 0.5 indicates strong hidden structural dependency. These files should likely be reviewed together during code review.

Hotspots

Files are classified by combining churn with author diversity:

  • Refactor Candidates: ≤1 author, ≥3 commits — single person iterating on a complex file
  • Watch List: >1 author — multiple people modifying the same code (coordination risk)

Circular Dependency Detection

Uses 3-color DFS (White/Gray/Black) to find all circular import chains:

  • White = unvisited, Gray = in current path, Black = fully explored
  • A Gray → Gray edge indicates a cycle
  • Reports all cycles, not just strongly connected components
Why 3-color DFS over Tarjan's algorithm? Tarjan finds SCCs (strongly connected components) but doesn't enumerate individual cycles. 3-color DFS is simpler to implement and directly reports all cycle paths, which is what developers need for debugging circular imports.

Impact Analysis

Uses BFS reverse traversal from the target symbol through the dependency graph. Each file in the blast radius receives a risk score 1–10 based on:

  • Depth from target (closer = higher risk)
  • Number of transitive dependents
  • Whether the file is in a critical path (high PageRank)

Ignored paths: vendor/, node_modules/, generated files, mock files.

Memory Retrieval at Query Time

Each user message triggers a hybrid search against embeddings.db:

  • Top 5 chunks retrieved (increased from 2 for better recall)
  • RRF delta threshold: 0.015 — chunks below this delta from the top score are dropped
  • Retrieved chunks are wrapped in <retrieved_memory> XML tags in the system prompt
  • Ordering: most relevant chunk placed LAST (Lost in the Middle optimization)

Session Summary Extraction

At session end, extractSessionSummary() produces a structured summary with:

  • Topics discussed (extracted from user messages)
  • Last substantive AI response
  • Key decisions made during the session

This summary is embedded and stored for future retrieval.

Inspiration: MemGPT — persistent memory architecture for LLM agents. Mantis adapts the tiered memory concept (working memory + long-term retrieval) to a CLI context where the "working memory" is the current REPL session and "long-term" is the .mantis/ directory.

Trade-offs & Decisions

DecisionAlternativeRationale
Ollama (local-first) Cloud APIs (OpenAI, Anthropic) Privacy: code never leaves the machine. $0 cost. Works offline. No API keys required for core features.
SQLite for everything PostgreSQL, Redis Zero dependencies. Embedded. Fast enough for single-user CLI. No server process. One file = easy backup.
Tree-sitter over LSP Language Server Protocol Multi-language from one library. No server process to manage. Faster cold start. Deterministic parsing.
RRF over learned fusion Learning-to-rank, neural re-ranker No training data needed. k=60 is robust across domains. Simpler. No model dependency for retrieval.
7 model tiers 2–3 tiers Latency optimization: ~40% of queries are trivial/fast (<300ms). Using a 70B model for "what is X?" wastes 3–5 seconds.
SEARCH/REPLACE edits Unified diff format Models produce cleaner output. No line-number precision required. Lower patch application error rate.
Parallel CODE+TESTS Sequential generation 2× wall-clock speedup. Stages are independent: tests don't need implementation to be written, only the plan.
Content-hash guard Always re-embed Startup re-indexing becomes O(changed) not O(all). Critical for large BRAIN.md files that rarely change entirely.
3-color DFS Tarjan's SCC algorithm Simpler implementation. Reports all individual cycles, not just SCCs. Developers need specific cycle paths to fix imports.
